Threads of Becoming vs. Other Frameworks

A Guide to Understanding When to Use What

Overview

Threads of Becoming is not intended to replace other valuable frameworks. Instead, it provides a diagnostic layer that helps you identify which frameworks to use when.

Think of it as a navigation system that sits beneath your existing practices, helping you recognize patterns before you intervene.

Detailed Comparisons

Threads vs. Polarity Management

What is Polarity Management?
Barry Johnson's framework for mapping interdependent pairs (e.g., centralization ↔ decentralization) and managing them over time.

How Threads is Different:

  • Diagnostic first: Threads helps you identify which polarity you're in before you start mapping
  • Universal patterns: Seven archetypal threads that appear across all contexts
  • Scalability: Maps individual, team, and organizational simultaneously
  • Embodied integration: Includes somatic practices for navigation

Use Both: Many organizations use Threads for diagnosis, then apply Polarity Management to specific threads. Example:

  1. Thread assessment identifies CONSENT (Yes ↔ No • Self ↔ Other) as primary tension
  2. Polarity mapping of "Centralized Control ↔ Distributed Autonomy"
  3. Ongoing polarity management with Threads language

Threads vs. Nonviolent Communication (NVC)

What is NVC?
Marshall Rosenberg's framework for empathetic communication focused on observations, feelings, needs, and requests.

How Threads is Different:

  • Organizational vs. interpersonal: Threads addresses systemic patterns, not just individual communication
  • Pattern recognition: NVC focuses on skills; Threads focuses on diagnosis
  • Complexity: Threads integrates multiple scales (individual, team, organizational)

Use Both: Individual conflicts (use NVC) often reveal thread collapses (use Threads to address systemic pattern). Example: Two leaders have needs conflict → reveals CONSENT thread collapse at organizational level.

Threads vs. Internal Family Systems (IFS)

What is IFS?
Richard Schwartz's therapy model focused on internal "parts" (manager, firefighter, exile) and accessing "Self" energy.

How Threads is Different:

  • External vs. internal: Threads maps organizational/relational tensions; IFS maps internal psychological parts
  • Collective patterns: Threads works at team/org level; IFS is individual therapeutic work

Use Both: Leaders' internal parts often mirror organizational threads. Example: Leader's "manager part" (control) vs. "exile part" (trust) reflects organizational CONSENT collapse. IFS for personal work; Threads for organizational application.

Threads vs. Agile/Scrum

What is Agile/Scrum?
Iterative software development methodology focused on sprints, retrospectives, and adaptive planning.

How Threads is Different:

  • Framework vs. diagnostic: Agile is a process; Threads helps diagnose why the process is struggling
  • Pattern naming: Threads explains why teams oscillate between over-planning and rushing

Use Both: Use Threads to diagnose collapse patterns in Agile teams. Example: Team oscillates between detailed sprint planning (PAUSE: Not Yet • More) and reactive firefighting (PAUSE: Now • Enough). Apply thread navigation to stabilize Agile practice.

Threads vs. OKRs (Objectives & Key Results)

What are OKRs?
Goal-setting framework popularized by Google, focused on ambitious objectives and measurable key results.

How Threads is Different:

  • Diagnosis vs. execution: OKRs set goals; Threads helps when goals conflict
  • Pattern recognition: Threads explains why OKRs feel impossible to achieve

Use Both: Set OKRs that honor both poles of a thread. Example: PAUSE thread → OKR: "Reduce decision-making time by 30% while maintaining quality standards"

Threads vs. Design Thinking

What is Design Thinking?
Human-centered innovation process (empathize, define, ideate, prototype, test) developed by IDEO and Stanford d.school.

How Threads is Different:

  • Problem vs. tension: Design Thinking assumes solvable problems; Threads addresses ongoing tensions
  • Diagnosis first: Threads helps identify if the challenge is actually a navigable tension, not a solvable problem

Use Both: Use Threads to identify which tensions are at play, then use Design Thinking to prototype solutions that honor both poles. Example: EMBODIMENT thread (Think ↔ Feel • Stay ↔ Go) → prototype dashboards that integrate both data and qualitative experience.

Threads vs. Theory U / Presencing

What is Theory U?
Otto Scharmer's framework for deep change through "presencing"—accessing the future that wants to emerge.

How Threads is Different:

  • Diagnostic precision: Threads names specific patterns; Theory U focuses on process of sensing
  • Immediate application: Threads provides actionable language now; Theory U requires longer journey

Use Both: Use Threads to name what's being sensed in presencing work. Example: Theory U retreat surfaces BECOMING thread (Same ↔ Different • Return ↔ Forward) → Threads provides language to navigate it.

Threads vs. Crucial Conversations

What are Crucial Conversations?
Patterson, Grenny, et al.'s framework for high-stakes dialogue when opinions vary, stakes are high, and emotions run strong.

How Threads is Different:

  • Systemic vs. situational: Threads addresses patterns; Crucial Conversations addresses specific moments
  • Diagnosis: Threads helps you understand which thread is driving the crucial conversation

Use Both: Use Threads to diagnose which thread is in collapse, then use Crucial Conversations skills to have dialogue about thread navigation. Example: CONSENT thread collapse → Crucial Conversation about decision rights.

Integration Decision Tree

Use this flowchart to decide which framework to apply:

START: You're facing organizational tension ↓ Q1: Do you know exactly which tension you're navigating? → YES: Go to Q2 → NO: Use Threads to diagnose ↓ Q2: Is it a specific interpersonal conflict? → YES: Use NVC or Crucial Conversations → NO: Go to Q3 ↓ Q3: Is it a known polarity you've already mapped? → YES: Use Polarity Management → NO: Use Threads to identify, then choose framework ↓ Q4: Do you need to design a solution? → YES: Use Design Thinking (while honoring thread) → NO: Use Threads navigation practices ↓ Q5: Is the tension internal (within individual)? → YES: Use IFS or personal work → NO: Continue with Threads at team/org level

When NOT to Use Threads

Threads is powerful, but not for everything. Don't use Threads when:

  1. You have a clear problem with a solution
    • Example: "Our software has a bug" → Fix the bug (not a thread). Threads is for tensions, not problems.
  2. You need a process/methodology
    • Example: "How do we run sprints?" → Use Agile (Threads doesn't provide process)
  3. The issue is individual skill-based
    • Example: "Manager needs feedback skills" → Use coaching/training (Threads doesn't teach skills)
  4. You want a decision-making tool
    • Example: "Should we pursue Strategy A or B?" → Use decision frameworks (Threads helps you hold the tension between them)
  5. It's an emergency requiring immediate action
    • Example: "System is down, customers are affected" → Fix it now, reflect on threads later

Summary: Threads as the Substrate

Think of Threads as the diagnostic and navigational layer beneath other practices:

  • Other frameworks → What to do
  • Threads → Why tensions arise and how to navigate them

You don't replace your existing toolbox—you add pattern recognition that helps you choose the right tool at the right time.

Recommended Learning Path

If you're new to these frameworks, here's a suggested learning path:

  1. Start with Threads → Learn pattern recognition
  2. Add NVC → Build communication skills for thread navigation
  3. Layer in Polarity Management → Deep-dive specific threads
  4. Apply Design Thinking → Prototype solutions that honor both poles
  5. Integrate Theory U → Access deeper emergence within threads
